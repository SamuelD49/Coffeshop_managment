# Coffee Shop Management System — Design Spec

**Date:** 2026-05-12
**Status:** Approved (brainstorming phase)
**Author:** Samuel

## Goal

A back-office management web app that replaces four paper forms used by a coffee shop today: Daily Sales Income, Purchase Requisition, Petty Cash Replenishment, and Payroll. Plus a structured employee record with HR documents and guarantors. Local-first (one PC at the shop, no internet required). A second-phase migration to Supabase is anticipated but out of scope for v1.

## Non-goals (explicit MVP scope cuts)

- No POS / customer-facing order screen (sales are entered manually at end of shift).
- No recipe-based inventory; no live stock levels or low-stock alerts.
- No supplier management, no customer/loyalty CRM.
- No automatic income tax bracket calculation — tax is typed manually per payroll entry.
- No multi-shop / multi-tenant; one database = one shop.
- No email/SMS, no password-reset-via-email, no 2FA.
- No rate limiting or brute-force protection (LAN-only deployment).
- No native mobile app — responsive web only.
- No i18n; English only, single currency, single date format.
- No cloud hosting; Supabase is v2.

## Architecture

A single Node.js process running on the shop's PC. SQLite database file lives alongside the app. Devices on the same WiFi reach it at `http://<shop-pc-ip>:3000`; the PC itself uses `localhost:3000`.

### Stack

- **Runtime:** Node.js + TypeScript
- **Web framework:** Express (classic MVC)
- **Views:** EJS server-rendered templates. HTMX for in-place updates on the sales-entry form (live totals).
- **DB:** SQLite via `better-sqlite3`
- **Migrations:** plain `.sql` files in `/migrations`, run on startup
- **Auth:** `express-session` with `connect-sqlite3` for persistence; `bcrypt` for password hashing
- **Styling:** Tailwind compiled to a single static CSS file
- **File uploads:** `multer` for receiving, `sharp` for thumbnails
- **Exports:** `exceljs` (Excel), `pdfkit` (PDF, including printable payslips)
- **Scheduled tasks:** `node-cron` (nightly DB backup)

### Project layout (MVC)

```
coffeeshop-mgmt/
├── /migrations            # numbered .sql files
├── /src
│   ├── app.ts             # express app + middleware composition
│   ├── server.ts          # entry point; runs migrations then listens
│   ├── /lib               # db, session, money, dates, uploads, audit
│   ├── /models            # one file per table, only place that touches the DB
│   ├── /controllers       # one per resource
│   ├── /middleware        # requireAuth, requireOwner, csrf, errorHandler
│   ├── /routes            # express routers wired to controllers
│   └── /views             # .ejs templates (layouts, partials, per-resource)
├── /public                # tailwind output, htmx.min.js, images
├── /data                  # gitignored — shop.db, uploads/, backups/
└── /tests                 # unit + integration
```

The `/models` layer is the only code that touches the database. This keeps the v2 Supabase migration contained.

## Data model

SQLite. All tables have `id INTEGER PRIMARY KEY`, `created_at`, `updated_at` (omitted below). Money stored as integer minor units (cents) to avoid float drift.

### `employees`

| Field | Notes |
|---|---|
| `full_name`, `phone` | Required |
| `national_id_number`, `national_id_type` | Kebele ID / Passport / Driver's License / other |
| `date_of_birth`, `gender`, `marital_status` | |
| `address` | Single free-text field |
| `emergency_contact_name`, `emergency_contact_phone`, `emergency_contact_relation` | |
| `position` | e.g. "Barista", "Cashier", "Manager" |
| `hire_date`, `termination_date` | `termination_date` nullable |
| `basic_salary` | Cents |
| `username`, `password_hash` | Nullable — non-login staff exist on payroll without logging in |
| `role` | enum: `owner` \| `employee` |
| `is_active` | Soft delete |
| `onboarding_status` | enum: `incomplete` \| `complete`; derived from required fields + required attachments + ≥1 guarantor |

### `guarantors`

Multiple per employee.

- `employee_id` FK
- `full_name`, `phone`, `address`, `relation_to_employee`
- `national_id_number`, `national_id_type`
- `occupation`, `workplace`
- `notes`

### `attachments` (polymorphic)

Single table for all file uploads.

- `owner_type` enum: `employee` \| `guarantor`
- `owner_id`
- `kind` enum: `profile_photo` \| `id_front` \| `id_back` \| `contract` \| `guarantor_letter` \| `other`
- `filename` (stored), `original_name`, `mime_type`, `size_bytes`
- `uploaded_at`, `uploaded_by` FK to `employees`

Physical files: `/data/uploads/{owner_type}/{owner_id}/{filename}`. Images get a `thumb_<filename>` sibling generated by `sharp`. Cascade deletion on owner removal is handled in application code (polymorphic, no FK).

### `menu_items`

- `name`, `price` (cents), `sort_order`, `is_active`

### `sales_sessions`

One per filled-out Daily Sales Income form = one employee's shift.

- `employee_id` FK
- `business_date` (date in shop timezone, adjusted by `business_day_cutoff`)
- `shift` (free text — "morning" / "evening" / etc.)
- `cash_amount`, `bank_transfer_amount` (cents)
- `notes`
- `status` enum: `open` \| `closed`

`total_amount` and `difference` are **not stored** — they are computed on read from line items + cash/bank fields (see Calculations).

### `sale_line_items`

Rows inside a sales session.

- `sales_session_id` FK
- `menu_item_id` FK
- `qty`, `unit_price_snapshot` (price at time of sale), `total` (cents), `remark`

### `purchase_requisitions`

Flat log — matches paper form.

- `purchase_date`, `description`, `unit`, `qty`, `unit_price`, `total`, `remark`
- `entered_by` FK to `employees`

### `petty_cash_entries`

- `entry_date`, `description`, `payer_name`, `amount` (cents)
- `type` enum: `expense` \| `refund` \| `replenishment`
- `remark`, `entered_by` FK

`cash_on_hand_after` is **not stored** — it's recomputed on every list render by walking entries ordered by `(entry_date, id)` (see Calculations).

### `payroll_runs`

One per month.

- `year`, `month`, `status` enum: `draft` \| `approved`
- `prepared_by` FK, `approved_by` FK (nullable)
- Unique constraint on `(year, month)`

### `payroll_entries`

One per employee per run. Pension rates are snapshotted so historical runs are immutable.

- `payroll_run_id` FK, `employee_id` FK
- `days_worked`, `basic_salary`
- `pension_employer_pct`, `pension_employee_pct` (snapshots)
- `pension_employer_amount`, `pension_employee_amount` (**stored** — denormalized snapshot)
- `gross_salary`, `income_tax` (manual), `advance_salary`, `total_deduction`, `net_payment` (**all stored**)
- `signed_at` (nullable)

Unlike sales sessions and petty cash, payroll entries store their computed totals. The reasoning: once a run is approved, it is immutable; stored values + snapshotted rates make the historical record self-contained and auditable.

### `settings`

Key/value table. Keys:

- Shop: `shop_name`, `shop_address`, `shop_phone`, `logo_path`
- Money: `currency_code` (default `ETB`), `currency_symbol` (default `Br`), `decimal_places` (`2`), `thousand_separator` (`,`), `decimal_separator` (`.`)
- Payroll: `pension_employer_default_pct` (`11`), `pension_employee_default_pct` (`7`), `standard_days_in_month` (`30`), `require_complete_hr_before_payroll` (`true`)
- System: `business_day_cutoff` (`00:00`), `timezone` (`Africa/Addis_Ababa`), `backup_path` (`./data/backups/`)

### `audit_log`

- `actor_id` FK, `action`, `entity`, `entity_id`, `at`
- No UI in v1; populated on all writes for forensics.

## Roles & authentication

Two roles: `owner` (full access) and `employee` (sales entry for own shifts only). Owner is a superset of employee — anything an employee can do, owner can do too.

### Login

- `POST /login` with `username` + `password`
- Lookup `employees` by username; reject if missing or `is_active=0`
- `bcrypt.compare` against `password_hash` (cost 12)
- Session payload: `{ employeeId, role }`, stored in SQLite via `connect-sqlite3`
- Cookie signed with `SESSION_SECRET` from `.env`

### Middlewares

- `requireAuth` — redirects to `/login` if no session
- `requireOwner` — 403 if `session.role !== 'owner'`

### Route protection

| Resource | Read | Write |
|---|---|---|
| Own sales session | self or owner | self (when `status=open`) or owner (always) |
| Other sales sessions | owner | owner |
| Menu, Employees, Purchases, Petty Cash, Payroll, Reports, Settings | owner | owner |
| Dashboard | both (content varies by role) | n/a |

### First-run bootstrap

If `employees` is empty on startup, all requests redirect to `/setup`. That page collects the first owner's name, username, password; creates the account; then `/setup` is disabled.

### Password handling

- bcrypt hashing on create and change
- Owner can reset any employee's password (no email flow)
- Each user can change their own password at `/account`

### CSRF & misc

- `csurf` middleware on all state-changing routes; hidden `_csrf` field in every form
- No rate limiting / lockout in v1 (LAN-only)
- All writes write a row to `audit_log`

## Pages & user flows

### Employee layout

Header (shop name, logout) + two buttons: **Start new shift**, **My past shifts**. That's it.

### Owner layout

Left sidebar:

> Dashboard → Sales → Menu → Employees → Purchases → Petty Cash → Payroll → Reports → Settings

Employees sits before Payroll to reflect the onboarding-first workflow.

### Page map

| Page | Path | Who | What |
|---|---|---|---|
| Login | `/login` | all | Username + password |
| Setup (first-run only) | `/setup` | unauthenticated | Creates first owner |
| Dashboard | `/` | both | Employee: shortcuts. Owner: today's totals — sales, cash vs bank, top items, purchases, petty cash spent |
| Sales — list | `/sales` | owner=all, emp=own | Filter by date range, employee, status |
| Sales — new | `/sales/new` | both | Pick shift label + date → creates row, redirects to entry page |
| Sales — entry | `/sales/:id` | own session or owner | Menu rendered as rows with qty inputs; live totals; cash/bank/difference/notes at bottom; "Close shift" |
| Menu | `/menu`, `/menu/new`, `/menu/:id` | owner | CRUD + reorder + activate/deactivate |
| Employees — list | `/employees` | owner | Status badge per row (Complete ✓ / Incomplete — N items missing) |
| Employee profile | `/employees/:id` | owner | Tabs: Personal, ID & Documents, Guarantors, Employment, Payroll history |
| Purchases | `/purchases` | owner | Date filter, inline add-row form, edit/delete |
| Petty cash | `/petty-cash` | owner | Same shape as purchases. Type selector. Running cash-on-hand displayed |
| Payroll — list | `/payroll` | owner | Monthly runs. "New run" button |
| Payroll — run | `/payroll/:id` | owner | Auto-populated table; edit days/tax/advance inline; "Approve" locks the run |
| Payroll — print | `/payroll/:id/print` | owner | Printable PDF matching paper sheet |
| Reports | `/reports` | owner | Tabs (Sales / Purchases / Petty Cash / Payroll). Date range, group-by, export Excel/PDF |
| Settings | `/settings` | owner | All keys above |
| Account | `/account` | both | Change own password + phone |

### Key flows

**Flow A — Cashier ends a shift.** Log in → "Start new shift" → pick shift label + date (defaults to today's business date) → land on entry page → type qty per menu item (totals update live via HTMX) → fill Cash, Bank transfer, Notes → "Close shift" sets `status=closed`. Employee can still view but not edit; owner can edit anytime.

**Flow B — Owner onboards an employee.** `/employees/new` → personal info → upload profile photo, ID front/back, signed contract → add ≥1 guarantor with their info and ID photos → set employment (position, hire date, basic salary, optional login credentials). Status badge flips to **Complete** when all required pieces are present.

**Flow C — Owner runs monthly payroll.** `/payroll` → "New run" → pick year+month. System creates a `payroll_run` and auto-creates one `payroll_entry` per active employee using their `basic_salary` and the default pension rates from Settings (snapshotted to the entry). Owner edits each row inline: days worked, income tax, advance. Pension amounts + gross + total deduction + net auto-compute. Save → status stays `draft`. **Approve** → status `approved`, run is locked, prepared_by/approved_by recorded. **Print** → PDF.

If `require_complete_hr_before_payroll=true` and any active employee's `onboarding_status=incomplete`, the "New run" flow warns and filters them out by default (with an override checkbox).

**Flow D — Owner reviews the day.** Dashboard cards: today's total sales, cash vs bank split, top 5 items, purchases entered today, petty cash spent. Each card drills into its underlying list.

**Flow E — Owner exports for accounting.** Reports → pick range → pick grouping → Export Excel / PDF.

## Calculation rules

All math runs server-side in `lib/money.ts`. Values stored as integer cents; formatted on output with `Intl.NumberFormat` using the currency settings.

### Sale line total

```
line.total = line.qty × line.unit_price_snapshot
```

### Sales session totals

```
session.subtotal     = sum(line.total) for all line items
session.total_amount = session.cash_amount + session.bank_transfer_amount
session.difference   = session.total_amount − session.subtotal
```

`difference` is **computed, not typed** (the paper form has the cashier write it; the system removes that opportunity for error/fraud). Negative = shortage, positive = overage; rendered red when non-zero. All three totals update live via HTMX on every qty/cash/bank input change.

### Petty cash — running cash-on-hand

Per-entry sign convention:

- `replenishment` → adds
- `refund` → adds
- `expense` → subtracts

```
cash_on_hand_after[n] = cash_on_hand_after[n−1] + signed_amount(entry[n])
```

Recomputed on every list render (single pass over entries ordered by `entry_date, id`). Editing or deleting a middle entry causes everything below it to recompute. The user never types `cash_on_hand_after`.

### Payroll math (per entry)

`pe_pct` and `pn_pct` are the snapshots taken at run creation.

```
gross_salary             = basic_salary × (days_worked / standard_days_in_month)
pension_employer_amount  = gross_salary × pe_pct / 100
pension_employee_amount  = gross_salary × pn_pct / 100
total_deduction          = pension_employee_amount + income_tax + advance_salary
net_payment              = gross_salary − total_deduction
```

- `income_tax` and `advance_salary` are typed by the owner — not computed.
- `pension_employer_amount` appears on the payslip and shop cost reports but does **not** reduce the employee's net pay.
- Run total row at the foot of the sheet sums every column.

### Rounding

Intermediate math stays in integer cents. Display rounding is half-up to `decimal_places`.

### Dates & "today"

- Timestamps stored as UTC ISO strings.
- "Today" in the UI = current date in shop timezone, adjusted by `business_day_cutoff`. A sale logged at 1 AM with `business_day_cutoff=04:00` files under yesterday's business date.
- `business_date` on a sales session is explicitly settable (defaults to today's business date), so the owner can back-fill a missed shift.

## Testing strategy

- **Unit tests:** `lib/money.ts`, `lib/dates.ts`, payroll calculation function. The math-critical pieces.
- **Integration tests:** one happy-path per resource (create sales session → add lines → close → totals match expectation).
- **Manual checklist** for everything else. This is an MVP for one shop.

## Deployment & operations

- `npm install && npm run build && npm start` on the shop PC.
- Optional: bundle with `pkg` so the owner double-clicks an icon and the server starts.
- LAN access: `http://<shop-pc-ip>:3000` from any device on the WiFi.
- **Backups:** `node-cron` nightly job copies `data/shop.db` → `data/backups/shop-YYYY-MM-DD.db`, prunes anything older than 30 days. Manual "Download backup" button in Settings.

## v2 — Supabase migration plan

Because `/src/models/*` is the only code that touches the DB, swapping backends is contained.

1. Provision Supabase. Port `migrations/*.sql` to Postgres (`INTEGER PRIMARY KEY` → `BIGSERIAL`, text-or-enum decisions per column).
2. Add `supabase-js` client in `lib/supabase.ts`.
3. Each model file gains a Supabase implementation. Select via env var `BACKEND=sqlite|supabase`.
4. Files migrate from `/data/uploads/` to Supabase Storage, same path scheme.
5. Auth: either migrate to Supabase Auth, or keep `express-session` + bcrypt and use Supabase only for data.
6. Optional hybrid: keep SQLite primary; small sync worker pushes deltas to Supabase on a schedule. Offline-tolerant.

Audit log, calculations, view layer, controllers don't change.

## Open questions

None blocking v1. The following will be revisited as v2 approaches:

- Whether to keep `express-session` or move to Supabase Auth.
- Whether the LAN deployment ever needs HTTPS (currently plaintext on local network).
- Whether income tax brackets become automatic in v2.
